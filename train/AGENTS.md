# 专列运营 ERP 静态原型：项目统一规范

> 本文件是本项目所有页面的强制设计与实现规范。每次新建、修改或补充 HTML 页面前，必须先读取并遵守本文件；同时读取 `PROJECT_IMPLEMENTATION_PLAN.md`，用于确认页面清单、模块分组、批次节奏和最终导航页范围。后续单页指令只补充当前页面的文件名、字段、假数据、行数和交互要求。若单页指令与本文件冲突，以用户最新、最明确的单页指令为准，其余要求继续沿用本文件。

## 1. 项目背景

- 项目名称：专列运营 ERP。
- 项目形态：HTML 静态原型。
- 业务说明：“专列”指旅游列车整列包销运营，示例列车为“昆明号”。
- 技术栈：纯静态 HTML + Tailwind CSS CDN + 项目共享 CSS/JS + 原生 JavaScript。
- 视觉基准：整体 UI 设计效果、布局密度、颜色气质和交互手感参照 `train_2/railtour-erp-static` 项目；当前项目已复制并适配其共享资产到 `assets/styles.css`、`assets/app.js`、`assets/favicon.svg`。
- 禁止使用 Vue、React、Angular、jQuery 或其他前端框架。

## 2. 文件与交付要求

- 每个页面生成一个独立的 `.html` 文件。
- 用户会在每个页面指令中指定文件名，必须严格使用该文件名，不得自行翻译、改写、添加前后缀或改变大小写。
- 每个 HTML 文件必须可直接在浏览器中独立打开并正常展示。
- 每个 HTML 文件必须包含：
  - `<!DOCTYPE html>`；
  - 完整的 `<html>`、`<head>` 和 `<body>`；
  - Tailwind CSS CDN 引入；
  - `<link rel="stylesheet" href="assets/styles.css">`；
  - `<script src="assets/app.js" defer></script>`；
  - 当前页面特有的少量原生 JavaScript，如共享 `assets/app.js` 已覆盖则不重复写。
- 页面依赖限于 Tailwind CDN 和当前项目 `assets/` 下的共享静态资产；不得依赖构建工具、包管理器、后端服务或其他本地页面才能显示主体内容。
- Modal、Drawer 及其遮罩层必须写在当前 HTML 文件内，默认隐藏，优先使用共享 `assets/app.js` 的 `data-modal-open`、`data-modal-close`、`.modal`、`.drawer` 机制控制显隐。
- 用户给出字段清单、示例数据和目标行数时，必须严格采用；数据不足时只按相同风格补齐至指定行数，不得擅自删除、合并或改名字段。

### 3.11 菜单收敛与入口形态（V1.1）

本项目从 V1.1 起区分“完整页面清单”和“左侧菜单主入口”。第 3 章的信息架构仍作为全量业务页面、面包屑和跳转关系的依据；但 Sidebar 不再承载全部 HTML 文件。

菜单收敛原则：

- 仅“列表页、看板页、工作台、分析页/报表页、配置总览页”保留在左侧菜单。
- “编辑、新建、详情、变更、确认、单据生成、导出、子配置、流程处理中间页”从左侧菜单移除。
- 从菜单移除的 `.html` 文件不得删除，统一视为“仅通过入口打开”的二级页面；如无独立 URL 必要，可将内容内联到主页面的 Modal 或 Drawer。
- 主页面需要通过顶部按钮、卡片按钮、表格行操作、详情抽屉按钮或流程步骤进入这些二级页面。
- `assets/app.js` 中的 Sidebar 菜单只注册保留入口；被移除页面仍可保留 `data-page`、面包屑和独立 HTML，以便通过链接直接打开和演示。

入口形态判定：

| 形态 | 适用场景 | 规范 |
| --- | --- | --- |
| Drawer 抽屉 | 内容少到中等、详情查看、轻量补充信息、无需复杂保存流程 | 右侧滑出，宽 `min(680px, 92vw)`，含标题栏、内容区、吸底操作栏；必须 `role="dialog"`、`aria-modal="true"`；支持 Esc 关闭、遮罩关闭、焦点圈定与关闭后焦点回退。 |
| Modal 弹窗 | 短确认、少字段新建/编辑、单步操作、轻量审核 | 居中，宽 `min(600px, 92vw)`，圆角 12px，遮罩 `rgba(5,19,31,.55)`；破坏性操作必须二次确认，且不可通过点击遮罩关闭。 |
| 二级页面 | 内容大、多步骤、多字段、需保存草稿、可视化强交互、需要独立演示 URL | 独立 `.html` 文件；顶部保留可点击返回主列表的面包屑、对象编号、状态标签；右上角动作区包含次操作、主操作和更多操作；返回主列表时尽量保留原筛选条件与滚动位置。 |
| 页内 Tab | 同一对象或同一数据集的不同视图 | 仅用于同对象内切换，不用于跨模块导航，也不得替代 Sidebar。 |

二级页面返回约定：

- 从列表或看板进入二级页面时，链接可带 `?from=页面文件名` 或 `?from=页面文件名#锚点`。
- 主列表页的查询、筛选和滚动位置，可使用 URL 参数、hash 或 `sessionStorage` 做静态原型级保留。
- 二级页面的面包屑第一个可返回项必须指向主入口页，而不是已从菜单移除的中间页。

## 4. 每页统一布局

内部后台页面统一采用 `train_2` 风格的后台管理系统布局。页面 `<body>` 必须设置 `data-page="文件名去掉 .html"`；共享 `assets/app.js` 会自动注入左侧双栏 Sidebar、顶部 Topbar、面包屑、当前班期选择、通知按钮和右侧用户“张经理”。

内部后台页的 HTML 结构建议如下：

```html
<body data-page="stock-traintype-list" data-title="车型模板">
  <header class="page-header">
    <div><h1>车型模板</h1></div>
    <div class="page-actions">...</div>
  </header>
  <section class="panel">...</section>
  <div class="modal" id="editModal" hidden>...</div>
</body>
```

1. 顶部 Topbar
   - 由共享 JS 注入，使用 `assets/styles.css` 中的 `.app-topbar`；
   - 左侧显示面包屑；
   - 右侧显示当前班期选择、通知按钮、头像和用户名“张经理”。
2. 左侧 Sidebar
   - 由共享 JS 注入，采用 train_2 的深海军蓝一级导航 + 浅色二级面板；
   - 一级菜单固定为：工作台、运力库存、产品定价、销售分销、订单交易、计调操作、财务结算、经营看板、系统管理；
   - 当前页面所属一级菜单和二级/三级菜单必须高亮；
   - 页面需要展示二级或三级导航时，名称和层级必须遵循“全站信息架构”；
   - Sidebar 仅显示 V1.1 菜单收敛后的主入口页，不再把编辑、新建、详情、变更、单据和子配置页作为菜单项展示。
3. 主内容区
   - 页面业务标题：`.page-header`；
   - 操作按钮区：`.page-actions`；
   - 筛选区：`.panel` + `.filters` 或 `.toolbar` + `.field`；
   - 数字卡片：`.metric-grid` + `.metric-card`；
   - 表格：`.panel` + `.table-wrap` + `.data-table`；
   - 表单：`.form-section` + `.form-grid` + `.field`；
   - 分页器：`.pagination`。

页面类型不适用某一结构时，可按用户的单页指令省略，但不得擅自省略列表页的必备结构。

C 端报名端页面使用 `<body data-shell="mobile">`，不注入后台 Sidebar/Topbar，保持 375px 手机原型布局；分销门户页面使用 `<body data-shell="channel" data-channel-page="d-home">`，由共享 JS 注入 PC 横向菜单。

## 5. 统一视觉规范

- 视觉最高基准：参照 `train_2/railtour-erp-static` 的 UI 效果，优先使用当前项目 `assets/styles.css` 中已经定义的类和 CSS 变量。
- 主框架色：海军蓝系 `--navy-950 #07192b`、`--navy-900 #0d2339`、`--navy-800 #173955`。
- 强调色：橙色 `--orange #e88b2b`、深橙 `--orange-dark #b9600f`、浅橙底 `--orange-soft #fff2e2`。
- 业务状态色：蓝 `--blue #3478e5`、绿 `--green #239665`、红 `--red #d84f58`、紫 `--purple #7758c4`，并使用对应 soft 背景。
- 页面背景：`--canvas #f3f6f8`；内容面：`--surface #ffffff`；分割线：`--line #e3e8ed`。
- 字体：使用系统中文字体栈；基础字号 14px，但后台组件保持 train_2 的紧凑字号层级，小标签、表格、按钮可使用 9-12px。
- 圆角：按钮 7px、卡片/面板 10px、Modal 12px；不额外做大圆角营销风格。
- 内容卡片使用 `.panel`，白底、浅边框、轻阴影，保持清爽、专业、信息密度较高的 ERP 后台风格。
- 表格使用 `.data-table`，表头浅底、行 hover、选中行左侧橙色强调；数字、日期、金额和操作列注意对齐。
- 业务状态统一使用 `.badge`，并按语义加 `.green`、`.blue`、`.orange`、`.red`、`.purple`。
- 主按钮优先使用 `.button.primary` 的深蓝样式；重要强调可用 `.button.orange`；次要操作用 `.button.secondary` 或 `.button.ghost`；危险操作用 `.button.danger`。
- 所有页面保持一致的 Topbar 高度、Sidebar 宽度、内容间距、按钮尺寸、表格样式和弹层风格。
- 默认以常见桌面管理端视口设计，同时避免普通宽度下出现明显遮挡或错位。

## 6. 页面与交互规则

### 6.1 列表页

列表页必须具备：

- 筛选栏；
- 操作区右上角“新建”主按钮；
- 数据表格；
- 表格末列“操作”；
- 操作项使用文字按钮；
- 底部分页器，至少显示“共 X 条”“每页 10 条”和翻页控件。

筛选、重置、分页、打开详情、新建和编辑等控件应提供适合静态原型演示的前端交互反馈；无需接入真实后端。

筛选交互优先使用共享机制：

- 搜索输入：`data-table-search="#tableId"`；
- 下拉筛选：`data-table-filter="#tableId"` + `data-filter-key="status"`，表格行写对应 `data-status="启用"`；
- 查询按钮：`data-apply-filters="#tableId"`；
- 重置按钮：`data-reset-filters`；
- 状态 Tab：`data-status-tabs data-target="#tableId"`，按钮写 `data-status="all"` 或具体状态值。

### 6.2 详情类操作

- 详情统一使用页面右侧 Drawer 展示。
- Drawer 使用 `<div class="modal drawer" id="xxxDrawer" hidden>`，默认隐藏，打开后带遮罩层。
- 必须支持关闭按钮、Esc 关闭、遮罩关闭、焦点圈定和关闭后焦点回退；如单页指令另有要求，以单页指令为准。
- Drawer 宽度遵循 `min(680px, 92vw)`，结构包含标题栏、内容区和吸底操作栏。
- Drawer 必须声明 `role="dialog"`、`aria-modal="true"`，并通过 `aria-labelledby` 或 `aria-label` 关联标题。

### 6.3 新建、编辑与确认类操作

- 新建、编辑、确认类操作统一使用居中 Modal。
- Modal 使用 `<div class="modal" id="xxxModal" hidden>`，默认隐藏，打开后带遮罩层。
- 打开按钮使用 `data-modal-open="xxxModal"`；关闭按钮使用 `data-modal-close`。
- 表单建议写 `data-success="保存成功"`，提交后共享 JS 自动展示 Toast 并关闭所在 Modal。
- 静态原型中的提交操作应有明确反馈，并合理关闭弹层或更新演示状态。
- Modal 宽度遵循 `min(600px, 92vw)`，圆角 12px，遮罩色为 `rgba(5,19,31,.55)`。
- 删除、释放、退款、作废等破坏性操作必须二次确认，且破坏性确认弹窗不可通过点击遮罩关闭。

Tab、视图切换和反馈优先使用共享机制：

- Tab：外层 `data-tabs data-scope="#scopeId"`，按钮 `data-tab="basic"`，内容 `data-tab-panel="basic"`；
- 视图切换：外层 `data-view-switch data-scope="#scopeId"`，按钮 `data-view="list"`，内容 `data-view-panel="list"`；
- Toast：按钮或链接使用 `data-toast="操作已完成"`；
- 二次确认：按钮使用 `data-confirm="确认释放库存？"`，可加 `data-confirm-success="库存已释放"`。

### 6.4 表单与反馈

- 表单标签、必填标记、占位提示和帮助文字需清晰统一。
- 禁用、只读、错误、成功等状态需有明显但克制的视觉区分。
- 删除、释放、退款、作废等高风险操作应有二次确认。
- 交互不得只停留在视觉摆设；用户指令涉及的主要按钮、Modal、Drawer、Tab、下拉或确认动作应可演示。

## 7. 数据与文案规则

- 严格使用用户在单页指令中给出的字段、示例文案、假数据和行数。
- 表格数据不足时，围绕“专列运营 ERP”和“昆明号”的业务语境补齐，风格保持一致。
- 不得使用与业务无关的占位文案，如无意义的 `Lorem ipsum`。
- 金额、日期、人数、库存、铺位、订单号和状态等数据格式在同一页面内必须一致。
- 面包屑、页面标题、Header 当前模块名和 Sidebar 高亮项必须互相匹配。

## 8. 每次生成页面前的执行清单

1. 确认用户指定的 HTML 文件名，并原样使用。
2. 确认页面所属一级模块、菜单层级、面包屑和高亮项。
3. 提取用户给出的字段清单、假数据、目标行数和交互要求。
4. 按本文件生成完整、可独立打开的 HTML。
5. 后台页 body 必须写正确的 `data-page`；分销门户写 `data-shell="channel"`；C 端写 `data-shell="mobile"`。
6. 将 Modal 和 Drawer 写入同一文件，默认隐藏，优先用共享 `assets/app.js` 的 `data-*` 交互驱动。
7. 检查列表页是否具备筛选、新建、操作列和完整分页信息。
8. 检查颜色、字号、圆角、卡片、表格和 Badge 是否符合 train_2 视觉。
9. 检查主要按钮与弹层交互是否可演示。
10. 检查页面是否严格遵循用户指定的字段、数据和行数。
11. 完成后在浏览器或等效渲染环境中检查布局与交互，修正明显的溢出、遮挡、错位或脚本错误。
